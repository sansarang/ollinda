# 순위진단(/api/rank-check) 진단·수정 기록

> 2026-07-11 · "실제 순위를 못 가져오고 추정 폴백만 돈다"는 리포트에 대한 진단·수정.

## TL;DR

**보고된 증상은 버그가 아니라 진단 프로브(내 curl 테스트)의 결함이었다.** 프로덕션에서 실제 상호로 호출하면 실측 순위가 정확히 나온다. 네이버 키·API·인증·매칭 모두 정상. 다만 스펙이 요구한 **상호 매칭 강건성**과 **진단 로깅**은 실질 가치가 있어 추가했다.

## 1단계 — 원인 진단 (실측)

프로덕션(https://ollinda.kr, Railway) 외부 프로빙으로 확정:

| 진단 항목 | 결과 | 근거 |
|---|---|---|
| 1. 네이버 키 로딩 | ✅ 정상 | `/api/place/search?q=스타벅스` → 실제 업체 5건 반환 = `NAVER_CLIENT_ID/SECRET` Railway에 설정됨 |
| 2. API·인증 | ✅ 정상 | 지역검색 API(`openapi.naver.com/v1/search/local.json`) 200, `X-Naver-Client-Id/Secret` 헤더 인증. **searchad(HMAC)와 분리 — 혼선 없음** |
| 3. 매칭 로직 | ✅ 작동(개선여지) | 실제 1위 업체명으로 호출 시 `rank:1` 정확 매칭. 공백만 제거해 지점명·특수문자엔 취약했음 |
| 4. 키워드 조합 | ✅ 정상 | "부산 동구 중고차" 등 [지역+업종] 롱테일이 5건 정상 반환 |

**실측 증거** (프로덕션 `--data-urlencode`로 정상 호출):
- `region=부산 동구, industry=중고차, name=미진대형자동차매매상사`(실제 1위) → **`rank:1, estimated:false`** ✅
- `name=ETF베이커리 부산역`(실제 1위 카페) → **`rank:1, estimated:false`** ✅
- `name=행복모터스`(가짜 상호) → **`rank:0, estimated:false`** ("상위 5위 밖", 정직) ✅

### 왜 처음엔 `rank:null, estimated:true`로 보였나 (오진의 원인)
이전 검증에서 내가 쓴 curl 프로브 자체가 잘못됐다:
1. `curl -d "industry=카페&region=부산"` — **상호(name) 없이** 호출 → diagnose가 name 없으면 의도적으로 `{"rank":None}` estimated 반환(정상 폴백 동작).
2. `curl -d "...region=부산 동구&name=행복모터스"` — `-d`로 **한글+공백을 URL 인코딩 없이** 전송 → 서버 form 파싱이 깨져 name 유실.
→ `--data-urlencode`로 제대로 보내면 정상. **앱 코드에는 문제가 없었다.**

## 2단계 — 수정 내용 (`app/services/place.py`)

버그는 없었으나 스펙이 요구한 두 개선을 안전하게 적용:

### ① 상호 매칭 강건성 — `_norm_name` / `_name_match`
- **정규화**: 공백·괄호·특수문자(`·・.,-–—_/&'"` 등) 제거 + 소문자화.
- **양방향 부분일치**: `내상호 ⊂ 네이버명` (일반) 또는 `네이버명 ⊂ 내상호`(네이버명 3자↑일 때만).
- **오탐 방지**: 내 상호 정규화 후 2자 미만이면 매칭 안 함.
- 흡수되는 케이스: `ETF베이커리`↔`ETF베이커리 부산역`, `루마썬팅`↔`초량 루마썬팅(본점)`, `스타벅스 강남`↔`스타벅스강남R점`.
- 의도적 미지원(보수): 중간 단어 생략(`미진자동차`→`미진대형자동차매매상사`) — 토큰 퍼지매칭은 결제 트리거에서 **가짜 "당신 상위!" 오탐** 위험이 커서 substring 보수 유지.

### ② 진단 로깅 — `search()` / `rank_detail()`
- `search`: 무키/빈쿼리 시 `configured` 로깅, non-200 시 **status·query·body** 경고 로깅(401/403=키, 429=레이트리밋, 결과밖 구분 가능).
- `rank_detail`: `kw·store·rank·top개수` info 로깅.
- → 앞으로 동일 증상 시 Railway 로그로 원인(키/레이트/매칭/범위밖)을 즉시 구분 가능.

## 정직성 준수 (유지)
- 실측 성공 → 실제 순위(1~5).
- 상위 5위 밖 → `rank:0` "상위 5위 안에 안 보여요" (가짜 순위 생성 안 함).
- 조회 불가(무키/에러/상호없음) → `estimated:true` "아직 상위에 없을 가능성" + "올린다로 진입" — **정직한 미노출 프레임**.
- **임의 X위 생성 절대 없음.**

## 남은 한계 (구조적 — 수정 불가)
- **네이버 지역검색 API는 `display` 최대 5** → 6위 이하 순위는 알 수 없음. 코드는 이를 `rank:0`("5위 밖")으로 정직 처리. 정밀 순위(예: 12위)는 이 공식 API로 불가능(스크래핑은 ToS 위반이라 안 함).
- 지역검색 정렬은 정확도순이라 개인화·거리 기반 실제 지도앱 순위와 다를 수 있음 — UI에 "참고용" 각주 유지.
- 실검색량은 `NAVER_SEARCHAD`(HMAC)로 별도 제공 중 — 순위와 무관하게 정상.

## 3차 개선 — 다중 키워드 스캔 + 손실 프레이밍 (신뢰도/결제 설득력)

### 문제
네이버 지역검색 API는 쿼리당 5건 → 소상공인 대부분(6위 밖)은 단일 쿼리로 "상위 5위 밖"만 떠서 결제 설득력이 없다.

### 해결 (`diagnose.py` 재작성)
1. **롱테일 다중 조회** (`_rank_keywords`): 넓은→좁은 [지역+업종] 3~4개 자동 생성(시 → 구 → 동/역 → 풀), 각각 `place.rank`로 상위5 조회. 5건 한계 안에서도 **여러 키워드를 커버**해 실측 순위를 최대한 많이 수집.
2. **실검색량 결합** (`searchad.keyword_volumes`, 이미 연동): 각 키워드 월 검색량을 붙임.
3. **정직한 미노출 프레이밍**: 5위 밖 키워드는 가짜 순위 없이 `missing`으로만 표기 + 검색량으로 **손실 프레이밍**("'부산 중고차'(월 8,000회)는 놓치는 중, 미노출 합계 월 9,200회").
4. **결과 구조**: `scan[{keyword,rank,volume,status}]` (status: top/missing/unknown), `caught`(실측 1~5위), `missing`(5위 밖+검색량), `missed_volume`(놓치는 총 검색량). CTA="놓치는 검색 월 N회 잡으러 가기". 하위호환 top-level 필드(`rank/keyword/headline/subline/cta`) 유지.
5. **랜딩 위젯**: 잡은 키워드(초록 ✅ N위) / 놓치는 키워드(주황 미노출+검색량)를 표로 렌더.

### 정직성 (재확인, 단위테스트 통과)
- `caught`만 실제 순위(1~5). `missing`은 전부 `rank:0`(가짜 순위 0건 검증).
- 조회불가(무키/네트워크)·상호없음 → `estimated:true` "추정" 명시.
- "무조건 1위" 보장 표현 없음 — "상위노출 시작/유지"까지만.
- 크롤링 없음 — 공식 지역검색 API 범위 내에서만.

### API 호출 비용
스캔당 place.rank 최대 4콜(Naver Local) + keyword_volumes 1콜(SearchAd). `MAX_SCAN=4`로 상한. (공개 엔드포인트 레이트리밋은 후속 과제)

## 검증
- 단위: `_name_match` 매칭 + 다중스캔(실측 1·3위 + 미노출 8천·1.2천회 혼합, missed_volume=9,200) + 엣지(전부미노출/조회실패/상호없음 → 가짜순위 0건) 통과.
- 통합: 스모크 4종 통과, `import app.main` 정상.
- 프로덕션: 배포 후 실제 상호로 다중스캔 확인(넓은=미노출·좁은=실측 혼합).
